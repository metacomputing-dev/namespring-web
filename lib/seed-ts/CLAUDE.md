 UI가 지금은 사용하지 않더라도 나중에는 사용하게 될 것이라, 전체적인 seed-ts 목적을 정확이      이해할 필요가 있음. 최종 작명된 이름 점수에 모두 활용되는 건 아니더라도(학파마다 다르므로), 기본 구조는 이름 조건(성은 
     한자입력까지 필수, 이름은 완전 비움, 부분 비움, 모두 채움 경우들이 가능) 안에서,  우선 사주 구성요소(용신 등)에 따라 가장 잘 
     맞는 최적의 이름 구조를 음양오행(에너지) - 한글, 한자, 사격, 사주 각각에 대해 계산하여 점수를 내면서 - 시뮬레이션을 통해 얻어       가능한 이름들을 추천하는 것이 기본이 되는 구조이고, 모든 가능한 경우의 수를 보고 싶을 때는 추천모드에서 전체모드로 바꾸면 됨. 
     (모두 채움의 경우, 이 때 단순 이름 평가가 됨). 이러한 플로우로 동작하도록 하고, 얻어진, 대상이 된 이름에 대한 점수 산정        
     근거(이것은 위 시뮬레이션에서 사용된 플로우를 한번 더 태우게 되는 상황), 사주를 포함한 성명학적 근거/해석, 추천요소 등이 
     얻어져서 리포트 생성 등등 원하는 서비스 구현을 위해 자유롭게 이용 가능한 형식으로 api return 해줘야 함. api의 입력은 음/양력
     태어난 시각 생년월일시분 성별 성(한자포함), 이름(글자수, 한글(초성중성종성) 및 한자(radical/부수, 옥편획수/원획수 등)),        
     위치/태어난 곳(위도, 경도), ...) 이런 정보가 될 것이고... api 입력 리턴은 saju-ts를 고려하여 saju-ts의 입력 출력을 함께        
     포괄해야 함(야자시 조자시 등등이 예). 출력도 마찬가지. 그러나 입출력이 유연하게 바뀌더라도 ui가 영향을 안 받는 방식이어야 함.  
     이러한 부분들을 정확히 숙지하여, 완벽하게 바로 동작하지 않는다 해도, 완벽한 모든 구성요소들이 포함되어 이를 내가 정교화 고도화 
     작업만 하면 바로 사용할 수 있는 가장 깔끔하고 완벽한 구조를 만들어서 api의 입력과 출력은 불변한 꼴이 되도록 하여 그 안에서 나는
      실제 seed-ts만 신경써서 일을 하면 되도록 해줘. ui작업자와 완전 독립적으로 서로 영향 주고받지 않으면서도 깔끔하게 일할 수 있게.
      내 요청사항을 철저히 숙지하여 최대한 많은 opus teammates를 생성하여 (10명 이상), plan mode를 다시 돌려서 가장 완벽한 seed-ts  
     구조를 갖춘 후(db는 지금 있는 거 바로 사용하는 형태로), 불필요한 파일이나 불필요한 요소들은 모두 삭제하고, 다 정리하고 여유가  
     되면 실제 유효한 함수, 수식, 규칙 등을 implememt하여 나의 수고를 덜어줄 수는 있겠지만, 명심해, 새상에서 가장 완벽한 구조를     
     갖추도록 하는 게 제 1의 임무야. 그렇게 진행해줘.
