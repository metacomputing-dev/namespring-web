## 3. 차이 분석

### 3.1 평가(최성수) — 거의 동일

- **종합 점수**: 58.0 vs 58.1 (0.1점 차이, 오차 범위)
- **사격수리**: 60.0 vs 60.0 (완전 동일)
- **사주 관련**: 45.0 vs 46.7 (1.7점 차이 — saju-ts 미연동 상태의 fallback 차이)
- **합격 판정**: 양쪽 모두 FAIL

### 3.2 이름 생성 — 구조적 차이 존재

| 비교 항목 | namespring-web | namesprint-w | 비고 |
|-----------|---------------|-------------|------|
| **총 후보 수** | 79,780 | 200 | MAX_CANDIDATES=200 제한 |
| **최고 점수** | 78.0 | 71.2 | 점수 스케일 차이 |
| **점수 범위** | 76~78 | 68~71 | namesprint-w가 약 7~8점 낮음 |
| **후보 한자** | 다양한 획수 (乎=5, 王=4, 玉=5 등) | 저획수 중심 (由=5, 人=2, 手=4 등) | 탐색 전략 차이 |
| **공통 이름** | 최호인, 최백인, 최민예 등 | 최유인, 최인수 등 | 일부 겹침 |

### 3.3 차이 원인 분석

1. **후보 수 제한 (200 vs 79,780)**
   - namesprint-w의 `MAX_CANDIDATES = 200`이 탐색 범위를 제한
   - namespring-web은 전체 유효 조합을 DFS로 탐색 후 모든 한자 조합을 시도

2. **한자 선택 전략**
   - namespring-web: 모든 한자를 사격수리 유효 조합에 대해 시도
   - namesprint-w: `findCharsForSlot()`에서 용신/기신 기반 필터링 후 4개씩만 선택
   - -> 더 정교한 한자 풀 + TopK 수집이 필요

3. **점수 스케일**
   - 평가에서는 거의 동일 (58.0 vs 58.1)
   - 생성에서 7~8점 차이는 한자 조합 풀의 차이에서 기인
   - namespring-web의 고득점 이름들이 namesprint-w의 탐색 범위에 포함되지 않음

4. **사주 통합**
   - namespring-web: saju-ts가 완전 통합되어 있어 사주 점수가 실제 값
   - namesprint-w: saju-ts 미연동으로 placeholder 사주 사용 (모든 후보에 동일한 46.7점)

## 4. 개선 권장 사항

### 즉시 개선 가능 (정교화 작업)
1. **MAX_CANDIDATES 증가** (200 → 2000+) 또는 TopK 히프 기반 수집으로 전환
2. **한자 조합 전략 확대**: `findCharsForSlot()`에서 4개 제한 해제, 전체 한자 풀 시도
3. **saju-ts 연동 완료**: 실제 사주 분석으로 용신/기신 반영 시 점수 차이 해소

### 구조적 개선
1. namespring-web처럼 `NameSearchService` 패턴 도입 (LimitedCollector + MinHeap TopK)
2. 전체 모드에서 모든 유효 조합에 대해 전수 탐색 후 TopK 수집
3. 이름 풀 품질 개선 (일상적이지 않은 한자 필터링)
